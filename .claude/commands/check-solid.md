# Проверка SOLID принципов

Проанализируй код проекта на соблюдение SOLID принципов:

## Что проверить:

### 1. Single Responsibility Principle (SRP)
- Каждый класс имеет только одну ответственность?
- Если класс изменится, будет ли только одна причина для изменения?
- Нет ли "божественных объектов" (God Objects)?

Проверь:
- Domain entities - содержат ли только бизнес-логику работы с собой?
- Use Cases - делают ли они только оркестрацию без бизнес-логики?
- Repositories - только работа с БД?
- Services - только специфичная логика?

### 2. Open/Closed Principle (OCP)
- Можно ли расширять функциональность без изменения существующего кода?
- Используются ли интерфейсы и абстракции?
- Есть ли возможность добавить новый способ взаимодействия (например, Discord бота) без изменения Use Cases?

Проверь:
- Можно ли добавить новую БД без изменения Domain Layer?
- Можно ли добавить новый Presentation Layer без изменения Application Layer?

### 3. Liskov Substitution Principle (LSP)
- Можно ли заменить реализацию интерфейса на другую без нарушения работы?
- Подклассы корректно реализуют контракты базовых классов?

Проверь:
- Реализации Repository могут заменять друг друга?
- Все implementation классы соответствуют своим интерфейсам?

### 4. Interface Segregation Principle (ISP)
- Не слишком ли "жирные" интерфейсы?
- Клиенты не зависят от методов, которые не используют?

Проверь:
- Repository интерфейсы - не слишком много методов?
- Use Cases получают только те зависимости, которые реально используют?

### 5. Dependency Inversion Principle (DIP)
- Зависимости направлены от конкретных реализаций к абстракциям?
- Domain Layer не зависит от Infrastructure Layer?
- Application Layer зависит от интерфейсов, а не реализаций?

Проверь:
- Domain Layer определяет интерфейсы репозиториев?
- Infrastructure реализует эти интерфейсы?
- Use Cases зависят от интерфейсов, а не конкретных классов?

## Формат ответа:

Для каждого принципа укажи:
- ✅ Соблюдается / ⚠️ Частично / ❌ Нарушается
- Примеры нарушений с указанием файлов и строк
- Рекомендации по исправлению

## Примеры проблем:

### ❌ Нарушение SRP:
```typescript
class TicketService {
  createTicket() { ... }
  sendEmail() { ... }      // Лишняя ответственность!
  saveToDatabase() { ... } // Лишняя ответственность!
}
```

### ❌ Нарушение DIP:
```typescript
class CreateTicketUseCase {
  // Зависимость от конкретной реализации!
  constructor(private repo: TicketRepositoryImpl) {}
}
```

### ✅ Правильно:
```typescript
class CreateTicketUseCase {
  // Зависимость от интерфейса!
  constructor(private repo: TicketRepository) {}
}
```

Начни проверку с анализа структуры проекта, затем проверь ключевые файлы в каждом слое.
