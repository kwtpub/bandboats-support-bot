Bandboats Support Bot - DDD Architecture

Введение

Это бот техподдержки, построенный на принципах Domain-Driven Design (DDD). Архитектура позволяет легко заменять способ взаимодействия с пользователем (Telegram Bot → Web → API) без изменения бизнес-логики.

Архитектура приложения

Приложение разделено на 4 слоя согласно DDD:

1. Domain Layer (Доменный слой)

Местоположение: src/domain/

Ответственность:

  • Бизнес-логика приложения
  • Сущности (Entities) и агрегаты (Aggregates)
  • Доменные сервисы (Domain Services)
  • Интерфейсы репозиториев (Repository Interfaces)
  • Доменные события (Domain Events)

Важно: Этот слой НЕ зависит от других слоёв и технологий!

См. также
Domain Layer Documentation

2. Application Layer (Слой приложения)

Местоположение: src/application/

Ответственность:

  • Сценарии использования (Use Cases)
  • Оркестрация операций
  • Управление транзакциями
  • DTO для входных/выходных данных

Не содержит: Бизнес-логику (она в Domain Layer)

См. также
Application Layer (Слой приложения)

3. Infrastructure Layer (Слой инфраструктуры)

Местоположение: src/infrastructure/

Ответственность:

  • Реализация репозиториев (работа с БД)
  • Интеграция с внешними сервисами
  • Работа с файловой системой
  • Email, SMS, Telegram API
См. также
Infrastructure Layer Documentation

4. Presentation Layer (Слой представления)

Местоположение: src/presentation/

Ответственность:

  • Интерфейсы взаимодействия с пользователем
  • Telegram Bot handlers
  • Web controllers (в будущем)
  • REST API endpoints (в будущем)

Важно: Разные презентационные слои используют одни и те же Use Cases!

См. также
Presentation Layer Documentation

Зависимости между слоями

Presentation Layer
↓
Application Layer
↓
Domain Layer
↑
Infrastructure Layer (реализует интерфейсы из Domain)

Правило: Зависимости направлены только ВНУТРЬ к Domain Layer. Domain Layer не зависит ни от чего!

Принципы разработки

SOLID принципы

  • **S**ingle Responsibility - один класс = одна ответственность
  • **O**pen/Closed - открыт для расширения, закрыт для модификации
  • **L**iskov Substitution - подтипы должны заменять базовые типы
  • **I**nterface Segregation - много специализированных интерфейсов
  • **D**ependency Inversion - зависимость от абстракций, а не реализаций

DDD паттерны

  • Entity - объект с уникальным идентификатором
  • Value Object - неизменяемый объект без идентификатора
  • Aggregate - группа связанных entities с корнем (Aggregate Root)
  • Repository - абстракция для доступа к хранилищу
  • Domain Service - бизнес-логика между несколькими aggregates
  • Use Case - сценарий использования системы

Начало работы

  1. Изучите структуру Domain Layer
  2. Посмотрите на Use Cases в Application Layer
  3. Изучите реализацию репозиториев в Infrastructure Layer
  4. Изучите Telegram Bot handlers в Presentation Layer
